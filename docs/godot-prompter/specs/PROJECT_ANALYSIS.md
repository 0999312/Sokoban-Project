# Sokoban 项目分析与产品梳理

> 文档目的：基于当前代码、现有 GDD、实施计划与运行时结构，对项目做一次“产品视角 + 技术视角”的现状总结，并整理出需要产品负责人确认的关键问题。
>
> 分析依据：`README.md`、`project.godot`、`docs/godot-prompter/specs/GDD.md`、`docs/godot-prompter/plans/IMPLEMENTATION.md` 以及核心运行时代码、编辑器、求解器、存档、输入、测试脚本。

---

## 1. 一句话结论

这是一个**已完成 v1.0 发布范围、进入维护阶段的 Godot 4.6 Sokoban 项目**，不是单纯玩法原型。

它已经具备：
- 完整的主流程：启动、主菜单、选关、游玩、通关结算、设置
- 完整的创作流程：关卡编辑、导入导出、分享码、求解验证、用户关卡游玩
- 完整的技术底座：存档、输入重绑、国际化、音频、辅助功能、后台求解器、自动化自检脚本

从产品形态看，它更接近“**可发布的开源推箱子游戏 + 内置轻量 UGC 工具**”。

---

## 2. 产品定位判断

## 2.1 当前产品定位

- 类型：经典 2D 推箱子游戏
- 差异点：支持多色箱子/多色目标点/中性槽，不是纯经典单规则复刻
- 扩展价值：内置关卡编辑器、XSB 导入导出、分享码、求解器验证
- 发布方式：免费开源
- 当前目标平台：以 PC 为主（Windows/macOS/Linux）
- 当前语言：简中、繁中、英文

## 2.2 当前版本的实际卖点

- 经典 Sokoban 规则，学习成本低
- 相比传统推箱子更有拓展性的“多色配对”机制
- 内置编辑器，玩家可自制内容
- 内置求解器，关卡可验证可解性与标准步数
- 已接入 257 关内容，内容量对独立开源项目来说已经很完整

## 3. 当前功能面梳理

## 3.1 玩家侧功能

- 启动画面与主菜单
- 章节化选关
- HUD：关卡名、步数、推数、计时
- 操作：移动、撤销、重做、重开、暂停
- 通关结算与星级评分
- 下一关衔接
- 设置面板：语言、音量、全屏、高对比度、减弱动画、键位重绑
- 自动存档与进度统计
- 键鼠/手柄切换与提示同步

## 3.2 创作侧功能

- 关卡编辑器
- 绘制工具：墙、地板、目标点、箱子、玩家、橡皮、单格/矩形/直线

- 测试游玩
- 导入：JSON / XSB / 分享码
- 导出：JSON / XSB / 分享码 / 缩略图
- 求解验证：异步运行、可取消、可把结果回写 metadata

## 3.3 内容侧功能

- W1：原创教学关
- W2：Microban 完整导入
- W3：XSB 精选导入
- 官方关卡与用户关卡统一索引

## 4. 技术特点总结

## 4.1 引擎与语言

- 引擎：Godot 4.6
- 主开发语言：GDScript
- 混合依赖：项目内存在少量 C# 插件代码，但业务主体是 GDScript
- 渲染方式：`Mobile` renderer
- 分辨率策略：`1280x720`，`canvas_items + expand`

这说明项目追求的是：
- 开发效率优先
- 2D 表现足够即可
- UI 与玩法层尽量轻量、稳定、跨平台

## 4.2 架构风格

项目采用的是**Autoload 服务层 + 核心模块分层 + 场景驱动 UI** 的 Godot 典型工程化结构。

主要特点：
- `GameState` 负责场景跳转与当前关卡状态
- `LevelLibrary` 负责官方/用户关卡索引
- `SaveManager` 负责版本化 JSON 存档
- `InputManager` 负责 gameplay 和 UI 的双输入体系
- `Board` 与 `Level` 负责纯玩法状态
- `BoardView` 负责视觉呈现
- `GameController` 负责把输入、玩法、HUD 串起来

这类结构的优点：
- 玩法逻辑与表现层解耦比较清晰
- 很适合继续扩内容、扩 UI、扩编辑器功能
- 测试和脚本化工具更容易做

## 4.3 重度依赖插件生态

项目并不是完全“原生裸写”，而是明显建立在插件/框架能力上：

- `mc_game_framework`
  - `EventBus`
  - `I18NManager`
  - `UIManager`
  - `RegistryManager`
- `guide`
  - 负责 gameplay 输入映射和重绑底座
- `sound_manager`
  - 负责音频总线、SFX/BGM 播放
- `gdfxr`
  - 用于部分音效资产生成/编辑生态

技术上这意味着：
- 项目复用度高，搭建速度快
- 但长期维护会受插件接口稳定性影响
- 新协作者需要先理解插件约定，而不只是 Godot 原生 API

## 4.4 玩法逻辑抽象做得比较干净

核心玩法没有直接耦合到场景节点里，而是做成了独立数据对象：

- `Level`：描述关卡静态数据
- `Board`：描述运行时状态
- `BoardCommand` / `UndoStack`：描述可逆操作

这让项目天然支持：
- Undo/Redo
- 求解器重放
- 编辑器测试游玩
- 分享码导入后直接验证
- Headless 自检脚本

这是项目里最有工程价值的一部分。

## 4.5 在经典 Sokoban 之上做了规则扩展

项目不是纯单色推箱子，而是已经落地了：

- 1..5 色箱子
- 1..5 色目标点
- 0 色中性槽
- 颜色匹配胜利判断
- 多色死格计算
- 多色求解启发式
- 编辑器中的多色绘制

这代表项目在“规则层”已经从复刻型产品走向了“可设计新谜题机制”的方向。

## 4.6 输入系统成熟度较高

输入层不是简单 InputMap，而是拆成两套：

- gameplay：走 GUIDE
- UI：走原生 `ui_*`

优点：
- Godot UI 导航能力不用重复造轮子
- gameplay 可做更细的重绑和设备识别
- 键盘/手柄提示可动态同步

这在独立项目里属于比较成熟的输入处理方式。

## 4.7 编辑器不是附属工具，而是正式子产品

从代码量、功能面和测试覆盖看，编辑器已经不是“辅助按钮”，而是一个完整系统：

- 独立场景
- 独立数据模型
- 导入导出管线
- 求解器验证闭环
- 用户关卡存储与主流程互通

从产品视角看，这个项目实际上包含两个面向用户的模块：
- 游戏本体
- 轻量关卡创作工具

## 4.8 求解器是项目的重要技术护城河

求解器目前具备：
- IDA*
- 死格剪枝
- 多色目标适配
- WorkerThreadPool 后台运行
- 解法展开为玩家步序列

它的作用不只是“炫技”，而是直接服务产品：
- 验证关卡可解
- 回写 `optimal_steps` / `optimal_pushes`
- 支撑评分体系
- 支撑编辑器创作闭环

这使得项目在“可持续产出新关卡”上更扎实。

## 4.9 存档、国际化、辅助功能都已落地

- 存档：JSON、版本迁移、原子写入
- 国际化：中简/中繁/英文，运行时切换
- 辅助功能：高对比度、减弱动画

这说明项目已经具备了发布态产品意识，而不是只关注玩法原型。

## 4.10 测试思路偏“可运行自检脚本”

项目有测试，但当前主力不是完整 GUT 套件，而是：

- `smoke_test.gd`
- `solver_test.gd`
- `solver_worker_test.gd`
- `editor_test.gd`
- `dialog_smoke.gd`

这类测试方式的特点：
- 接入成本低
- 适合 Godot headless 场景
- 对当前项目足够实用

但也意味着：
- 测试组织还偏工程实用型
- 距离标准化持续集成测试体系仍有提升空间

## 5. 当前产品结构判断

## 5.1 双主循环

这个项目实际存在两条完整用户循环：

### A. 游玩循环

启动 → 主菜单 → 选关 → 游玩 → 通关结算 → 下一关/返回选关

### B. 创作循环

主菜单 → 编辑器 → 绘制关卡 → 验证/测试 → 保存/导出/分享 → 回到“我的关卡”游玩

这意味着它不是单线产品，而是“**玩法产品 + UGC 工具产品**”的组合。

## 5.2 目标用户初步判断

- 经典益智解谜玩家
- 喜欢做关卡的创作者型玩家
- Sokoban / XSB 社区受众
- 喜欢开源、小而完整单机项目的玩家

## 6. 当前实现与文档的对齐情况

大方向上，代码与文档是对齐的，但有少量值得注意的差异。

## 6.1 已基本对齐的部分

- v1.0 已完成、进入维护阶段
- 257 关内容已接入
- 编辑器链路已完整打通
- 多色机制已落地
- 输入重绑、音频、i18n、辅助功能已落地

## 6.2 发现的轻微差异

### 差异 A：本地化资源格式

GDD 中写的是 CSV，但当前 `Boot` 实际加载的是 `locale/*.json`。

结论：
- 实现以 JSON 为准
- 文档可能还保留了较早阶段的描述

### 差异 B：测试体系描述

GDD 中写“单元测试：GUT”，但当前仓库里的主要自动化验证是 headless 脚本。

结论：
- 当前项目有测试能力
- 但“GUT 已成为主测试框架”这件事并未完全体现在仓库结构里

### 差异 C：存档版本描述

GDD 示例里是 `version: 1`，当前代码中的 `SaveManager` 已经是 `CURRENT_VERSION = 2`。

结论：
- 这是实现正常演进
- 文档示例需要视作概念性示例，不应完全等同当前实现

## 9. 待确认问题清单

以下问题建议作为产品确认会的输入。全部用提问形式整理，便于逐项答复。

### 9.1 产品方向

1. 你希望这个项目未来主要被定义为“推箱子游戏”，还是“带编辑器的推箱子创作平台”？
2. 当前 v1.0 完成后，你的下一阶段重点是继续补内容，还是补发布能力（导出、商店页、宣传素材、CI）？
3. 你是否希望“多色箱子”成为本项目的核心品牌差异点，而不是只作为进阶机制存在？

### 9.2 平台与发布

4. 当前文档写的是 PC 平台，但项目启用了 `Mobile` renderer；你的真实发布目标是否仍然只限桌面端？
5. 你是否计划在近期加入 Steam 发布准备，例如成就、云存档校验、商店资源、启动参数与导出脚本？
6. 你是否打算在 v1.1 或更后续版本中认真推进 Steam Workshop，而不是只保留分享码？

### 9.3 玩法与内容

7. 257 关是否已经满足你对首发内容量的预期，还是你仍希望首发前继续补更多原创关？
8. W1/W2/W3 的章节结构是否已经定稿，还是你还想调整章节顺序、命名或难度曲线？
9. 你是否希望未来的官方关卡以“经典单色”为主，还是逐步把多色/中性槽作为主流设计方向？
10. 对评分体系来说，你是否接受部分关卡的 `optimal_steps` 仍然是人工估计，而不是全部由求解器回填？

### 9.4 编辑器与 UGC

11. 你希望编辑器更多服务“你自己做官方内容”，还是也要明确面向玩家开放并降低使用门槛？
12. 你是否希望后续补一个更明显的“新手编辑引导”，例如模板、教学、错误解释或示例关卡？
13. 现在的分享码已经能满足传播需求了吗，还是你希望后面接入更直接的文件分享、二维码或在线仓库？

### 9.5 技术与维护

14. 你是否希望把当前 headless 自检脚本继续作为主测试方式，还是要逐步补齐 GUT/gdUnit4 体系？
15. 本地化资源你希望最终统一成 JSON，还是希望回到 CSV/PO 一类更适合翻译协作的格式？
16. 你是否准备继续依赖现有插件栈，还是希望未来逐步减少对 `mc_game_framework` / `guide` / `sound_manager` 的外部耦合？
17. 求解器目前是 GDScript 版本；如果后续出现大关卡性能瓶颈，你是否接受把求解器单独迁到 C#？

### 9.6 产品表现

18. 你是否满意当前“像素关卡素材 + 扁平矢量 UI”的混合风格，还是未来希望进一步统一美术语言？
19. 当前辅助功能只有高对比度与减弱动画；你是否还计划补色盲友好、屏幕阅读描述或更强的输入辅助？
20. 主菜单、HUD、通关动画目前已经能用；你是否还想继续做一轮更明显的视觉 polish？
